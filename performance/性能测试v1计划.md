# styleBI k6 性能测试计划 v1（简易版）

## 一、当前配置

```yaml
CPU: 16核
堆内存: 4GB
堆外内存: 6.4GB
节点: 1台
问题：4GB内存太小，16核CPU发挥不出来

建议：Heap调到12GB（如无法调整，并发数需砍半）

二、测试项清单
编号	名称	测什么	怎么测	通过标准
T1	冒烟测试	脚本能不能跑	1个用户打开报表	无报错
T2	冷加载	第一次打开要等多久	1/50/100人分别测	P95 < 3秒(简单表)
T3	联动响应	点筛选后要等多久	20/80/120人测联动	P95 < 2秒
T4	容量测试	多少人会挤爆	50→100→200→300→500人	找到拐点
T5	混合测试	挂着不动的人占不占资源	总300人(150活跃+150非活跃)	活跃用户响应不恶化
T7	数据量测试	数据多了会不会卡	固定50人，换1M/10M数据	1M正常，10M不崩
T8	组合测试	数据多+人多一起上	50人+1M数据	不崩就行
三、快速执行命令
bash
# 1. 准备用户（200个）
k6 run -e USER_COUNT=200 dist/setup-users.js

# 2. T1 冒烟
k6 run -e USERS=1 dist/test-d1.js

# 3. T2 冷加载（50人）
k6 run -e USERS=50 -e VSNAME=Dashboard_Medium dist/test-d1.js

# 4. T3 联动（80人）
k6 run -e USERS=80 -e VSNAME=Dashboard_Medium dist/test-d1.js

# 5. T4 容量测试（逐步加压）
k6 run -e USERS=100 dist/test-d1.js
k6 run -e USERS=200 dist/test-d1.js
k6 run -e USERS=300 dist/test-d1.js

# 6. T5 混合测试（150活跃+150非活跃）
k6 run -e USERS=300 -e INACTIVECOUNT=150 dist/test-d3.js

# 7. T7 数据量测试
k6 run -e USERS=50 -e VSNAME=Dashboard_1M dist/test-e1.js

# 8. T8 组合测试
k6 run -e USERS=50 -e VSNAME=Dashboard_1M dist/test-e2.js

# 9. 清理用户
k6 run -e USER_COUNT=200 dist/cleanup-users.js
四、执行顺序
天数	做什么
第1天	调内存→导数据→建报表→创建用户→跑T1冒烟
第2天	T2冷加载 → T3联动
第3天	T4容量 → T5混合
第4天	T7数据量 → T8组合
第5天	汇总报告 → 清理
五、记录模板
测试项	用户数	数据量	P95响应	错误率	是否拐点
T2	50	基础	2.1秒	0%	否
T4	200	基础	3.5秒	0%	否
T4	300	基础	8秒	2%	是
六、停止条件（满足任一即停）
响应时间突然翻倍

错误率 > 1%

服务CPU > 90%

内存使用 > 85%

七、如果内存只有4GB（不调整）

T2	50/100	20/50/80
T3	80/120	30/50
T4	500上限	150上限
T5	300	120
T7/T8	50人	20人
bash
# 4GB内存下的保守命令
k6 run -e USERS=80 dist/test-d1.js        # T2上限80
k6 run -e USERS=50 dist/test-d3.js        # T5上限120总用户

八、一句话总结
先调内存到12GB，然后按顺序跑T1→T2→T3→T4→T5→T7→T8，记录每个阶段的P95响应时间和错误率，找到响应时间突然飙升的那个用户数就是容量拐点。
