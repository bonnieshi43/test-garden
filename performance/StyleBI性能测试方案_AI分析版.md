# StyleBI 产品终极性能测试方案

## 第一部分：核心测试模型与抽象 (The "What")

本方案将 StyleBI 抽象为**三层混合负载系统**，所有测试均基于此模型设计。

### 1. 负载类型定义

| 负载类型 | 典型操作 | 技术特点 | 核心瓶颈 |
| :--- | :--- | :--- | :--- |
| **Type A: HTTP API** | 登录、Dashboard 初始化、资源树加载、EM管理 | 短连接、无/弱状态 | DB查询、缓存命中、接口并发 |
| **Type B: WebSocket** | 筛选器联动、下钻、图表刷洗、实时数据推送 | **长连接、强状态、服务端推送** | 连接数、消息队列堆积、Fan-out开销、CPU序列化 |
| **Type C: 前端渲染** | Dashboard渲染、大数据图表绘制、交互反馈 | 浏览器CPU/GPU密集 | DOM操作、图表库性能、数据量大小 |

### 2. 关键耦合关系（必须测试）

- **A → B**：打开Dashboard（HTTP）→ 建立WS连接 → 开始交互。风险点：初始化慢，Token不一致。
- **B → C**：筛选器操作（WS）→ 返回数据 → 前端渲染。风险点：WS快但前端卡顿，数据量过大导致掉帧。
- **B ↔ A**：WS操作触发后端计算，可能调用其他API/DB。风险点：WS和API争抢CPU/DB资源。

### 3. 用户与数据模型（测试参数化）

- **用户画像**：Viewer (70%, 低频操作)，Analyst (25%, 高频筛选)，Admin (5%, 管理API)。
- **数据量级**：小数据 (1k行)，中数据 (100k行)，**大数据 (1M行，关键)**。
- **并发模型**：阶梯式加压 100 → 500 → 1000 → 3000 并发用户。
- **测试时长**：短压 (10分钟，找极限)，长稳 (4-8小时，测内存泄漏)。

## 第二部分：分层测试策略与执行 (The "How")

### 🔴 最高优先级：核心BI交互（类型 A+B+C 耦合）

| 测试场景 | 测试目的 | 工具组合 | 关键指标 |
| :--- | :--- | :--- | :--- |
| **场景1: Dashboard冷启动** | 模拟Viewer打开含20+组件的复杂仪表盘 | k6 (HTTP) + Playwright | 端到端加载时间 (P95 < 3s)、JS Bundle大小、首屏FCP/LCP |
| **场景2: 高频筛选器联动** | **核心场景**。模拟Analyst高频触发filter/brush/drill | k6 (WebSocket) + Playwright | **WS消息延迟(P95 < 500ms)**、前端渲染帧率(FPS > 30)、组件联动端到端延迟 |
| **场景3: 混合负载极限** | **最关键场景**。同时模拟70% Viewer (HTTP) + 25% Analyst (WS) | k6 (混合脚本) | 系统整体吞吐量(TPS)、资源竞争下的P99延迟、错误率 (< 1%) |
| **场景4: 实时推送压力** | 高频数据变更下，STOMP推送吞吐量 | k6 (WebSocket) | 1000+ WS连接建立时间、消息吞吐量与丢包率、断线重连恢复时间 |

### 🟠 高优先级：数据引擎与后端（类型 A 深挖）

| 测试领域 | 具体场景 | 测试工具 | 关键指标 |
| :--- | :--- | :--- | :--- |
| **查询引擎** | JDBC查询百万行、MongoDB聚合、跨源JOIN | k6 (HTTP) + JMeter | 响应时间、吞吐量、内存峰值、CPU占用 |
| **物化视图(MV)** | 千万行数据MV构建、50用户并发命中同一MV | k6 (HTTP) | MV构建耗时、查询加速比、锁等待时间 |
| **图表/报表引擎** | 10万+点图表渲染、1000页PDF导出 | k6 (HTTP) + Locust | 服务端渲染时间、导出耗时、队列等待时间 |
| **Ignite集群** | 节点故障转移、`@ClusterProxy`路由开销、10GB缓存读写 | 混沌工程 + k6 | RTO、额外延迟百分比、IOPS |

### 🟡 中优先级：管理系统与多租户（类型 A 业务流）

| 测试领域 | 具体场景 | 测试工具 | 关键指标 |
| :--- | :--- | :--- | :--- |
| **REST API并发** | 500并发请求Portal接口、批量用户/权限管理 | Locust (业务流程) | P95响应时间 (< 300ms)、数据库事务耗时 |
| **多租户隔离** | 100个组织同时活跃，`orgId`缓存键查找 | k6 (HTTP) | 性能衰减比、上下文切换开销 |
| **数据连接器** | 连接池耗尽、外部API限速重试 | k6 (HTTP) | 连接获取等待时间、重试成功率 |
| **存储层(Blob)** | 100MB报表资产并发读写 | k6 (HTTP) | 读写吞吐量(MB/s)、锁竞争时间 |

### 🟢 低优先级：安全与辅助功能

| 测试领域 | 具体场景 | 测试工具 | 关键指标 |
| :--- | :--- | :--- | :--- |
| **安全认证** | 500用户SSO登录风暴、高频API权限校验 | k6 (HTTP) | 认证吞吐量、单次权限检查耗时 |
| **前端内存泄漏** | 长时间使用Portal/EM，反复打开关闭Dashboard | Playwright + Chrome DevTools | 长时使用后内存无增长、无detached DOM树 |

## 第三部分：推荐工具链 (The "Tooling")

### 最终工具组合方案

| 负载类型 | 推荐工具 | 理由 |
| :--- | :--- | :--- |
| **WebSocket 基准与混合** | **k6** | 原生支持WS、轻量、场景化脚本强、资源占用低，可同时模拟HTTP+WS |
| **HTTP API 基准** | **k6** | 高并发能力远超JMeter，脚本简单 |
| **复杂业务流程 (EM管理)** | **Locust** | Python编写复杂逻辑方便，Web UI实时监控直观 |
| **前端用户体验验证** | **Playwright** | 可模拟真实浏览器，测量FCP/LCP/FPS，检测内存泄漏 |

## 第四部分：执行步骤 (The "When")

### Step 1: 环境与数据准备（第1周）

- 部署独立测试环境。
- 生成 1k, 100k, 1M 三档数据集。
- 准备包含 20+ 组件的标准测试 Dashboard。

### Step 2: 单负载基准测试（第2周）

- 使用 k6 压测 **HTTP API** (Login, Tree, Search)，找出 QPS 极限与 DB 瓶颈。
- 使用 k6 压测 **WebSocket**，找出单节点最大连接数与消息吞吐极限。
- 使用 Playwright 测试 **前端**，记录不同数据量下的 FCP/LCP。

### Step 3: 核心耦合场景压测（第3周）⭐

- 执行 **k6 混合脚本** (70% HTTP Viewer + 25% WS Analyst)，监控资源竞争下的 P99 延迟。
- 执行 **Playwright + k6 联合**，在 k6 加压的同时，用 Playwright 验证 UI 响应是否掉帧。

### Step 4: 后端深度与稳定性测试（第4周）

- 使用 Locust 执行复杂 EM 业务流程压测。
- 使用 k6 执行物化视图、报表导出、Ignite 集群故障转移等专项测试。
- 执行 4-8 小时 **长稳测试**，监控内存/WS连接是否稳定。

### Step 5: 结果分析与调优（第5周）

- 汇总所有指标，对照阈值（如 P95 < 500ms, CPU < 70%）。
- 定位瓶颈点：慢 SQL、Ignite 路由开销、前端渲染函数、WS 消息队列堆积。

## 第五部分：量化指标体系 (The "Goal")

| 类别 | 指标 | 目标阈值 (P95) | 监控工具 |
| :--- | :--- | :--- | :--- |
| **WebSocket** | 消息端到端延迟 | < 500ms | k6, Grafana |
| | 单节点最大连接数 | > 2000 | Prometheus |
| | 消息吞吐量 | > 5000 msg/s | k6 |
| **HTTP API** | 响应时间 (Dashboard加载) | < 3s | k6, Loki |
| | 错误率 | < 1% | k6 |
| **系统资源** | CPU利用率 | < 70% | Prometheus |
| | 内存占用 (24h) | 无持续增长 | Prometheus + pprof |
| **前端** | 首屏加载时间 (FCP) | < 1.8s | Playwright, Lighthouse |
| | 大数据表格渲染帧率 | > 30 FPS | Chrome DevTools |
| | 长时间使用内存泄漏 | 无 | Chrome Memory Profiler |

---

## 第六部分：最终结论

> **StyleBI 系统的真实性能瓶颈，99% 的概率出现在“WebSocket 持续交互 + 大数据量 + 多用户并发”的叠加场景中。**
>
> **因此，本方案的核心测试策略是：以 k6 为核心，统一压测 HTTP + WebSocket 混合负载；用 Locust 覆盖管理类长业务流程；用 Playwright 闭环验证前端用户体验。**
