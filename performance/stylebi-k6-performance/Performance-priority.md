# StyleBI Performance Priority

> File name kept as requested: `Perfoance-priority.md`
> Scope: prioritize what this repository should test first
> Repository: `stylebi-k6-testing`

---

## 1. Goal

领导已经把测试目标重新拆成两类：

1. **Performance 测试**
  - 单一性能测试
  - 关注单个 Dashboard / 单类操作 / 大数据量下的数据加载与渲染
2. **并发测试**
  - 多用户访问多个 Dashboard
  - 关注不同用户量、不同操作组合下的稳定性与容量

这个仓库当前最强的能力，不是做底层 JVM 或 MV 内部机制分析，而是做：

- Viewsheet 打开性能
- 筛选联动性能
- WebSocket / STOMP 交互性能
- 数据量变化对 Dashboard 的影响
- 多用户并发访问 Dashboard 的稳定性与容量

所以优先级需要围绕 **“先把最容易落地、最能暴露真实问题的场景跑起来”** 来排。

---

## 2. Priority 分级定义


| 级别     | 含义       | 处理原则               |
| ------ | -------- | ------------------ |
| **P0** | 现在就必须开始  | 没有这部分，整体测试无法形成有效基线 |
| **P1** | 第一阶段必须完成 | 能最快暴露核心性能问题        |
| **P2** | 第二阶段完成   | 直接补充第一阶段结论，帮助定位边界  |
| **P3** | 第三阶段完成   | 对容量模型和复杂场景有帮助      |
| **P4** | 后续扩展     | 当前仓库可部分支持，但不是最先做   |
| **P5** | 暂缓       | 当前仓库不适合作为第一实现载体    |


---

## 3. 总体结论

### 最应该先开始的方向

**P0 / P1 应优先从 `Performance 测试` 开始。**

原因：

1. 单一性能测试更容易形成稳定基线
2. 更容易区分是“数据量问题”还是“并发问题”
3. 当前仓库已经有现成脚本：`test-d1`、`test-d2`、`test-e1`
4. 如果单用户 / 小用户量下性能已经差，并发测试结论会失真

### 顺序建议

1. **先做 Performance 基线**
2. **再做并发容量测试**
3. **最后再做混合场景和边界验证**

## P0：必须先做的基础性能测试

### P0-1 Dashboard 打开性能基线

**类型**：Performance

**为什么是 P0**：

- 这是最核心的用户体验指标
- 当前仓库已经直接记录 `viewsheet_open_time`
- 后续所有测试都需要拿它做对照

**要回答的问题**：

- 单个 Dashboard 打开到底要多久？
- 简单 / 中等 / 复杂 Dashboard 差异多大？
- 在 large VS 下，首次打开耗时是否已经不可接受？

**建议场景**：

- Simple Dashboard
- Medium Dashboard
- Large / Complex Dashboard
- 数据量：100K / 1M / 10M
- 用户量：1 / 5 / 10

**对应脚本**：

- `dist/test-d1.js`
- `dist/test-d2.js`（复杂度增强版）

---

### P0-2 Dashboard 筛选联动性能基线

**类型**：Performance

**为什么是 P0**：

- 打开只是第一步，真正影响用户的是交互过程
- 当前仓库已经直接记录 `selection_time`
- 最能暴露“数据加载 + 计算 + 渲染”的真实成本

**要回答的问题**：

- 在 large VS 上做一次选择操作要多久？
- 清除筛选再刷新要多久？
- 数据量上去后是否出现非线性变慢？

**建议场景**：

- 1 个选择器 + 1~4 个图表联动
- 100K / 1M / 10M 数据
- 用户量：1 / 5 / 10

**对应脚本**：

- `dist/test-d1.js`
- `dist/test-d2.js`
- `dist/test-e1.js`

---

## P1：第一阶段必须完成的高价值测试

### P1-1 大数据量 Performance 测试

**类型**：Performance

**为什么是 P1**：

- 领导明确提到要在 large VS 上做操作
- 这是产品侧最关心的单体性能问题
- 当前仓库已经有 `E1`、`E2` 的基础

**要回答的问题**：

- 当数据从 100K 增长到 1M / 10M，打开和联动的曲线怎么变？
- 哪个量级开始明显恶化？
- 是“打开慢”还是“联动慢”更明显？

**建议场景**：

- `Dashboard_100K`
- `Dashboard_1M`
- `Dashboard_10M`
- 操作：打开、筛选、清空筛选、刷新

**对应脚本**：

- `dist/test-e1.js`

---

### P1-2 Dashboard 复杂度 Performance 测试

**类型**：Performance

**为什么是 P1**：

- 同样数据量下，Dashboard 复杂度本身会导致性能差异
- 当前仓库已经具备 2 图 / 3 图 / 4 图场景
- 能帮助判断是“数据量问题”还是“Dashboard 设计复杂度问题”

**要回答的问题**：

- 2 图和 4 图的差异有多大？
- 多图表、多选择器联动会不会触发性能跳变？
- large VS 慢，是因为数据大还是因为组件太多？

**建议场景**：

- 2 图 Dashboard
- 3 图 Dashboard
- 4 图 Dashboard
- 4 图 + 多选择器 Dashboard

**对应脚本**：

- `dist/test-d2.js`

---

### P1-3 基础并发容量测试

**类型**：并发测试

**为什么是 P1**：

- 在有单体基线以后，第一件事就是确认容量边界
- 当前仓库对这块支持最好
- 最容易回答“系统现在能撑多少用户”

**要回答的问题**：

- 同时打开并操作一个 Dashboard，系统能稳定支持多少用户？
- 50 / 100 / 200 / 300 用户下，响应时间如何变化？
- 错误率何时开始上升？

**建议场景**：

- 1 个标准 Dashboard
- 用户量：50 / 100 / 200 / 300 / 500
- 每个用户做相同操作：打开、筛选、清空、刷新

**对应脚本**：

- `dist/test-d1.js`

---

## P2：第二阶段应该做的测试

### P2-1 多 Dashboard 并发测试

**类型**：并发测试

**为什么是 P2**：

- 领导明确提到“多个用户访问几个 Dashboard”
- 这比单 Dashboard 并发更接近真实情况
- 但前提是先知道单 Dashboard 的基线和容量

**要回答的问题**：

- 多个 Dashboard 混合访问时，系统是否出现明显抖动？
- 是否有某个复杂 Dashboard 把整体性能拖慢？

**建议场景**：

- 3 个 Dashboard：Simple / Medium / Large
- 用户分布：50% / 30% / 20%
- 操作组合：打开、筛选、清除、刷新

**当前仓库支持程度**：

- **部分支持**
- 目前可以通过环境变量和多个脚本分组跑
- 后续最好增加“多 Dashboard 随机路由”脚本

---

### P2-2 Active / Inactive 用户混合并发测试

**类型**：并发测试

**为什么是 P2**：

- 真实系统里不是所有用户都持续操作
- 这个场景能更接近在线运行状态
- 当前仓库已经支持

**要回答的问题**：

- 活跃用户和挂起用户共存时，系统资源消耗如何？
- 仅保持 Dashboard 打开状态，会不会大量占用资源？

**建议场景**：

- 总用户量：100 / 200 / 300 / 500
- 其中 30%~80% 为 inactive

**对应脚本**：

- `dist/test-d3.js`

---

### P2-3 大数据量 + 并发组合测试

**类型**：并发测试

**为什么是 P2**：

- 这是最接近生产压力的场景之一
- 但必须在单体性能和基础并发都清楚后再做

**要回答的问题**：

- 1M / 10M 数据下，100 用户并发时会发生什么？
- 大数据量和并发叠加时，拐点在哪？

**建议场景**：

- 1M 数据 + 50 / 100 用户
- 10M 数据 + 50 / 100 用户

**对应脚本**：

- `dist/test-e2.js`

## 6. 一句话建议

### 从 AI 分析角度，最应该先开始的是：

**P0：先做单体 Performance 基线，尤其是 large VS 的打开与联动性能。**

原因很简单：

- 如果 large VS 单用户都慢，并发测试只会放大问题，不能帮助判断根因；
- 先把 Performance 曲线拉出来，再做并发容量，结论才清晰。
