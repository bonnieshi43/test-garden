# 浏览器性能测试指南：Network 与 Performance 面板配合使用

## 一、核心分工


| 面板              | 负责回答的问题             | 适用场景                |
| --------------- | ------------------- | ------------------- |
| **Network**     | 资源什么时候开始下载、花了多久、谁慢了 | 加载慢、接口超时、资源未缓存、并发阻塞 |
| **Performance** | 资源到达后浏览器做了什么、为什么卡   | 交互卡顿、动画掉帧、长任务、布局抖动  |


> **一句话总结：Network 看“等得久”，Performance 看“动得慢”。**

---

## 二、测试流程：先 Network，后 Performance

### 第一步：先用 Network 排除网络问题

**操作流程：**

1. 打开 DevTools → **Network** 面板
2. 勾选 `Disable cache`（测首次加载时）
3. 勾选 `Preserve log`（防止跳转清空）
4. 复现问题 → 观察瀑布流

**判断标准：**


| 看到的现象                | 结论         | 下一步                  |
| -------------------- | ---------- | -------------------- |
| 某个请求 TTFB > 500ms    | 服务器慢       | 后端优化，不用看 Performance |
| 资源 `Size` 大且未压缩      | 未启用 gzip   | 开启压缩                 |
| 相同 URL 出现多次          | 无缓存 / 重复请求 | 配置 `Cache-Control`   |
| 大量请求排队 (`stalled` 长) | 并发限制       | 域名拆分或升级 HTTP/2       |
| 所有请求都很快，但页面仍慢        | **问题不在网络** | 切到 Performance       |


> ✅ 如果 Network 没发现明显瓶颈，再去 Performance。

---

### 第二步：再用 Performance 分析运行时

**操作流程：**

1. 切换到 **Performance** 面板
2. 设置 CPU 节流为 `4x` 或 `6x`（放大问题）
3. 点击录制 → 复现问题操作（3-5秒）→ 停止
4. 按顺序分析：FPS → CPU 条 → 主线程火焰图 → Call Tree

**判断标准：**


| 看到的现象          | 结论        | 定位方法                       |
| -------------- | --------- | -------------------------- |
| FPS 出现红条       | 掉帧        | 看红条区域主线程在做什么               |
| 主线程大面积**黄色**   | JS 执行过重   | Call Tree 找自己的耗时函数         |
| 主线程大面积**紫色**   | 布局/样式计算频繁 | 找 `Layout` 事件，看是否 `forced` |
| 主线程大面积**绿色**   | 绘制过重      | 看 Layers，是否重绘区域过大          |
| 主线程有大段空白（Idle） | 主线程空闲     | 问题可能在网络或 GPU，回 Network 查   |


> ✅ Performance 定位到具体代码行后，去 Sources 面板优化。

---

## 三、配合使用的典型场景

### 场景1：页面加载慢

Network 先看：
├── HTML 的 TTFB 是否过长？ → 后端问题
├── 关键 CSS/JS 是否被阻塞？ → 调整加载顺序
├── 有无未压缩的大文件？ → 开启压缩
└── 如果 Network 正常 → 切 Performance

Performance 再看：
├── 首次绘制（FP）之前主线程在做什么？
├── 是否有长任务阻塞了解析？
└── 是否有不必要的脚本执行？

### 场景2：交互卡顿（点击/滚动/动画）

Performance 直接看（Network 基本无关）：
├── FPS 红条出现在哪个操作时刻
├── 主线程是黄色（JS）还是紫色（Layout）
├── Call Tree 定位到具体函数
└── 修复后重新录制对比

Network 只在一种情况看：
└── 卡顿时 NET 条是否密集（说明在发请求）→ 可能请求阻塞了主线程

### 场景3：页面越用越卡（内存问题）

Performance 先看：
├── 录制时勾选 Memory
├── JS Heap 折线是否持续上涨不回落
└── 确认有内存泄漏 → 切到 Memory 面板

Network 不看（内存问题与网络无关）

## 四、快速决策表


| 用户原话       | 首选面板                        | 什么时候切到另一个                  |
| ---------- | --------------------------- | -------------------------- |
| “打开要好几秒”   | **Network**                 | 网络都正常但页面还是白屏 → Performance |
| “滚动一卡一卡的”  | **Performance**             | 几乎不用切                      |
| “点按钮没反应”   | **Performance**             | 如果看到大量 NET 请求 → Network    |
| “图片一张张蹦出来” | **Network**                 | 图片加载完还卡 → Performance      |
| “用久了就变卡”   | **Performance** (勾选 Memory) | 确认内存泄漏后 → Memory 面板        |


## 五、Performance 面板分析技巧

### 录制前准备

- ☑ 开启隐身模式（禁用扩展干扰）
- ☑ 关闭其他标签页
- ☑ 切换到 Performance 面板
- ☑ 拉宽 DevTools 窗口，确保看到 CPU 节流下拉框
- ☑ 勾选 Screenshots
- ☑ 设置 CPU 节流为 `6x slowdown`

### 分析顺序（从宏观到微观）

1. **看 Overview 区的 CPU 条**：判断问题类型（黄色=JS、紫色=Layout、绿色=绘制）
2. **看主线程火焰图**：找宽度超过 50ms 的长任务
3. **点击长任务 → Call Tree**：展开找自己的代码文件，双击跳转
4. **看 Screenshots**：将卡顿与画面变化对应

### 各颜色含义


| 颜色     | 含义      | 问题方向              |
| ------ | ------- | ----------------- |
| **黄色** | JS 脚本执行 | 去 Call Tree 找耗时函数 |
| **紫色** | 布局/样式计算 | 可能存在强制同步布局        |
| **绿色** | 绘制      | 重绘区域过大，考虑提升图层     |
| **浅蓝** | 网络请求    | 去 Network 面板查看    |

## 六、Network 面板分析技巧

### 关键列说明


| 列名          | 含义       | 关注点                   |
| ----------- | -------- | --------------------- |
| `Waterfall` | 请求时间线    | 看是否有长条（慢请求）           |
| `Size`      | 资源大小     | 原始大小 vs 传输大小，差距小说明未压缩 |
| `Time`      | 总耗时      | 点击查看各阶段分解             |
| `Status`    | HTTP 状态码 | 304=缓存有效，404/500=错误   |


### 耗时分解（点击某个请求查看）


| 阶段                   | 含义      | 问题归属           |
| -------------------- | ------- | -------------- |
| `Stalled/Blocked`    | 排队等待    | 客户端（并发限制）      |
| `DNS Lookup`         | 域名解析    | DNS 服务器        |
| `Initial connection` | TCP 握手  | 网络/服务器         |
| `SSL`                | TLS 握手  | 服务器配置          |
| `Request sent`       | 发送请求    | 通常很快           |
| `Waiting (TTFB)`     | 等待服务器响应 | **服务器**（最常见瓶颈） |
| `Content Download`   | 下载内容    | 网络带宽           |


---

## 七、实战口诀

> **录制要短（3-5秒），先看颜色（CPU条），再找长块（火焰图），最后钻代码（Call Tree）。**

> **Network 查“来的慢不慢”，Performance 查“跑得顺不顺”。**

---

## 八、常见问题速查


| 问题     | 查看位置                    | 典型特征                     |
| ------ | ----------------------- | ------------------------ |
| 服务器响应慢 | Network → TTFB          | TTFB > 500ms             |
| 资源未压缩  | Network → Size          | 传输大小 ≈ 原始大小              |
| 缓存未生效  | Network → Size          | 显示实际大小，非 `(disk cache)`  |
| JS 执行慢 | Performance → Call Tree | 黄色长条，找到自己的函数             |
| 强制同步布局 | Performance → Layout    | 紫色条块，Summary 显示 `forced` |
| 内存泄漏   | Performance → Memory 折线 | 阶梯式上涨不回落                 |
| 并发请求阻塞 | Network → Waterfall     | 请求排队，`stalled` 长         |
| 重绘区域过大 | Performance → Layers    | Paint 绿色区域远超元素范围         |
