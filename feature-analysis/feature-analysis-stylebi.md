# 使用场景

适用于：
- 新功能（Feature）开发 PR
- 功能增强或流程变更
- 可能影响数据流、权限、报表渲染逻辑的改动

不适用于：
- 单纯 Bug 修复（请使用 bug-analysis 模板）
- 依赖升级或纯代码重构


# Prompt 模板

你是一个资深测试架构师，熟悉企业级 BI / 报表分析系统（如 StyleBI）的大型功能测试方法。

请访问 [PR_URL]，仔细阅读 PR 的：
- 标题
- 描述
- 相关 issue（如果有）
- 代码变更（diff）

该 PR 属于一个“新功能（Feature）”开发。

请优先识别本次变更的风险等级（低 / 中 / 高），并根据风险等级决定测试覆盖深度。

请从完整功能测试的角度出发，生成一份结构化、系统化的测试建议，覆盖：
- 功能行为
- 数据流与状态变化
- 权限控制
- 异常处理
- 回归风险

在涉及认证机制、权限上下文、Token 处理、会话管理或分布式请求逻辑时，才展开 Stateless Session 相关专项分析；如无相关改动，请明确说明无需此类测试。

若涉及功能新增、删除、重命名或用户流程变更，应设计产品文档、在线帮助、API 文档及 Release Note 的一致性验证测试；若无功能层变化，可说明无需文档验证。

请使用中文输出，并包含以下六个部分：

## 1. Feature 功能理解与范围界定
- 总结核心目标
- 业务问题说明
- 涉及模块识别
- 类型判断（UI / 数据 / 性能 / 架构）

## 2. 测试策略设计
包含但不限于：
- 功能正确性
- 数据流与状态路径
- 权限验证
- 边界与异常处理
- 性能影响评估
- 兼容性测试
- 并发风险（如适用）

说明每类测试的动机。

## 3. 详细测试用例设计
每个用例必须包含：
- Case
- Type
- Pre-condition
- Steps
- Expected result

必须覆盖：
- 正向流程
- 负向流程
- 极端情况
- 不同数据规模（如涉及计算）

## 4. 数据影响与一致性验证
- 数据结构变化
- 持久化影响
- 历史数据兼容性
- 报表/分析逻辑影响

## 5. 回归风险分析
评估是否影响：
- 报表渲染
- 数据刷新
- 权限体系
- 导出功能
- Scheduler / Materialized Views / API / Storage

给出回归测试建议。

## 6. 本地化与可用性分析
若涉及：
- UI 页面变更
- Label 或提示信息变化

则说明是否需要进行本地化测试；
否则明确说明无需相关测试。
