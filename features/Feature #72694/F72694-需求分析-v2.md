## 一、需求分析（Requirement Analysis）

### 1. 功能理解与范围

**功能核心目标**
当图表的某条轴存在多个绑定（multi-binding）时，坐标轴标签可能在视觉上被分离，造成阅读困难。此功能允许用户通过 UI 操作（右键轴标签 → Properties）将轴标签移至对侧，而无需手动编写 Script 调用 `AxisSpec.setAxisStyle()`。

**解决的业务问题**
消除多绑定轴场景下标签分散的视觉问题，降低用户操作门槛（从 Script 配置降为 GUI 一键勾选）。

**涉及模块**
| 模块 | 文件 |
|---|---|
| UI（前端） | `axis-label-pane.component.html`、`axis-label-panel-model.ts` |
| 对话框逻辑 | `axis-property-dialog.spec.ts`（测试桩）、`AxisPropertyDialogModel.java` |
| 数据模型传输 | `AxisLabelPaneModel.java`、`AxisDescriptor.java` |
| 图渲染核心 | `AxisSpec.java`、`GraphGenerator.java` |
| 本地化资源 | `srinter.properties` |

**功能类型**：UI + 数据层（全栈跨层特性）

---

## 二、实现分析（Implementation Analysis）

### 1. 改动类型

**Feature**，影响层级：UI + 业务逻辑 + 数据层（跨层）

**受影响的系统行为 / 用户路径**
- 所有图表的轴属性对话框（portal 侧）
- 通过 Script 调用 `AxisSpec.getAxisStyle()` 的自定义脚本
- 已有 Dashboard 从 XML 反序列化图表配置的路径
- 图表渲染引擎（`GraphGenerator.setupAxisSpec` → `AxisSpec`）

---

### 2. 需求实现一致性

**实现完整的部分**
- 全栈数据流打通：UI checkbox → `AxisLabelPaneModel` → `AxisPropertyDialogModel` → `AxisDescriptor` → `GraphGenerator` → `AxisSpec`，链路完整。
- XML 序列化 / 反序列化同步（`writeAttributes` + `parseAttributes`）。
- `equals` / `hashCode` / `clone` 均已更新，对象状态管理完整。
- 本地化资源（英文）已添加。

**实现不足**

1. **仅添加了英文 locale**（`srinter.properties`）。其他语言的 `.properties` 文件（如中文、日文等）未更新，非英文用户界面会显示 key 原文或乱码。 🔴 **测试-分析**：会单独自动生成其他版本的文件
2. **`getAxisStyle()` 语义变更未在文档/注释中警告**：该方法现在是有副作用的动态计算，而非纯 getter，这对调用 `getAxisStyle()` 并做 `style & AXIS_LABEL_OPPOSITE_SIDE` 位操作的外部脚本会产生隐式影响。
3. **没有 EM（Enterprise Manager）侧的处理**：PR 只改动了 `portal` 的 `axis-label-pane.component.html`，未确认 EM 侧是否存在相同的 axis-label-pane；若存在，则 EM 中该功能无法使用。 🔴 **测试-分析**：不需要, 进portal端chart的功能

**隐式行为变化**

- `getAxisStyle()` 从纯 getter 变为动态计算方法，任何调用该方法的测试或脚本若期望获取"原始 style 值"将得到错误结果。

---

### 3. 关键实现风险

**风险 1：`getAxisStyle()` 语义变更导致 Script API 双重叠加**
- **来源**：`getAxisStyle()` 现在动态 OR `0x10`，而 `AXIS_SINGLE2 = 0x2 | 0x10`，`AXIS_DOUBLE2 = AXIS_DOUBLE | 0x10`。
- **影响路径**：用户通过 Script 写 `chart.getAxis("...").setAxisStyle(AxisSpec.AXIS_SINGLE2)` 后，再通过 UI 勾选 checkbox，`getAxisStyle()` 返回的值中 `0x10` 已被计入两次（`style` 字段含 `0x10`，方法再次 OR）；若调用者用返回值做 `setAxisStyle(getAxisStyle())`，会形成自我强化循环，但最终 bit 值不变（幂等），风险等级较低但语义混乱。
- **潜在后果**：外部脚本使用 `getAxisStyle()` 判断当前是否为"对侧"状态时，结果不可靠。

🔴 **测试-分析**：不存在, script的优先级高于UI, ui设置无效.

**风险 2：旧有通过 Script 配置 `AXIS_SINGLE2`/`AXIS_DOUBLE2` 的 Dashboard 加载后行为**
- **来源**：旧 Dashboard XML 中 `style` 字段可能已包含 `0x10`（通过 Script `setAxisStyle(AXIS_SINGLE2)` 持久化），而 `labelOnSecondaryAxis` 默认为 `false`。
- **影响路径**：反序列化后 `labelOnSecondaryAxis=false`，但 `style` 中仍含 `0x10`，UI checkbox 显示为"未勾选"，但渲染实际仍在对侧——UI 与渲染状态不同步，用户界面误导。
- **潜在后果**：用户看到 checkbox 未勾选，但标签实际在对侧，混淆用户预期。

🔴 **测试-分析**：不存在, script的优先级高于UI, ui设置无效.

**风险 3：EM 侧 axis-label-pane 未处理**
- **来源**：PR 仅修改了 `web/projects/portal/src/app/graph/dialog/axis-label-pane.component.html`。
- **影响路径**：若 EM 中存在轴属性对话框（可能复用同一组件或有独立实现），该功能将在 EM 中无法使用或行为不一致。
- **潜在后果**：功能覆盖不完整，管理员无法通过 EM 配置。

🔴 **测试-分析**：不存在, em端不涉及本功能
