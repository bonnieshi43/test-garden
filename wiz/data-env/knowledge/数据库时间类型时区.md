# 数据库时间类型时区

---

# 一、先给你一个核心结论（非常重要）

👉 数据库时间类型本质分三类：

| 类型 | 是否存时区 | 代表含义 |
| --- | --- | --- |
| ✅ 带时区（timezone-aware） | ✔ | 绝对时间（全球唯一时刻） |
| ⚠️ 会转换但不存原始时区 | 半支持 | 存UTC或session时区 |
| ❌ 不带时区（naive） | ✖ | 只是“本地时间字符串” |

---

# 二、主流数据库对比（重点）

## 🟢 1. PostgreSQL（最标准，推荐模型）

### 类型：

| 类型 | 是否支持时区 |
| --- | --- |
| `timestamp with time zone`（timestamptz） | ✅ |
| `timestamp without time zone` | ❌ |

---

### ✅ `timestamptz`（重点）

```sql
INSERT INTO t VALUES ('2024-04-13 10:00:00+02');

```

👉 实际存储：

```text
2024-04-13 08:00:00 UTC

```

✔ 自动转 UTC  
❌ 不保存原始 +02

---

### 🚨 特点

*   存的是 **绝对时间**
    
*   查询时按 session timezone 展示
    

---

## 🟡 2. MySQL

### 类型：

| 类型 | 是否支持时区 |
| --- | --- |
| `TIMESTAMP` | ⚠️（自动转换） |
| `DATETIME` | ❌ |

---

### ⚠️ `TIMESTAMP`

```sql
INSERT INTO t VALUES ('2024-04-13 10:00:00');

```

👉 行为：

*   写入：按 session timezone → 转 UTC 存
    
*   读取：UTC → 转 session timezone
    

✔ 有时区感知  
❌ 不存原始时区

---

### ❌ `DATETIME`

```sql
2024-04-13 10:00:00

```

👉 就是字符串

❗没有任何时区概念

---

## 🟣 3. BigQuery

### 类型：

| 类型 | 是否支持时区 |
| --- | --- |
| `TIMESTAMP` | ✅（UTC） |
| `DATETIME` | ❌ |
| `DATE` | ❌ |

---

### ✅ `TIMESTAMP`

👉 **永远是 UTC**

```sql
TIMESTAMP("2024-04-13 10:00:00+02")
→ 2024-04-13 08:00:00 UTC

```

✔ 绝对时间  
❌ 不存原始时区

---

### ❌ `DATETIME`

```sql
2024-04-13 10:00:00

```

👉 没有时区

---

## 🔵 4. Snowflake（最容易踩坑）

### 类型：

| 类型 | 是否支持时区 |
| --- | --- |
| `TIMESTAMP_TZ` | ✅ |
| `TIMESTAMP_LTZ` | ⚠️ |
| `TIMESTAMP_NTZ` | ❌ |

---

### ✅ `TIMESTAMP_TZ`

```sql
2024-04-13 10:00:00 +02:00

```

✔ 存时间 + 时区

---

### ⚠️ `TIMESTAMP_LTZ`（默认）

👉 行为：

*   存 UTC
    
*   显示按 session timezone
    

✔ 类似 PostgreSQL timestamptz  
❌ 不存原始时区

---

### ❌ `TIMESTAMP_NTZ`

👉 无时区

---

## 🔴 5. Oracle

### 类型：

| 类型 | 是否支持时区 |
| --- | --- |
| `TIMESTAMP WITH TIME ZONE` | ✅ |
| `TIMESTAMP WITH LOCAL TIME ZONE` | ⚠️ |
| `DATE` | ❌（但有时间部分） |

---

### ⚠️ `DATE`（非常坑）

```sql
DATE '2024-04-13'

```

👉 实际：

```text
2024-04-13 00:00:00

```

❌ 有时间但没时区

---

## ⚫ 6. SQL Server

### 类型：

| 类型 | 是否支持时区 |
| --- | --- |
| `datetimeoffset` | ✅ |
| `datetime` | ❌ |

---

### ✅ `datetimeoffset`

```sql
2024-04-13 10:00:00 +02:00

```

✔ 真正带时区

---

# 三、总结对比表（精华版）

| 数据库 | 真正支持时区 | 默认安全类型 |
| --- | --- | --- |
| PostgreSQL | ✅ | timestamptz |
| MySQL | ⚠️ | timestamp |
| BigQuery | ✅（UTC） | timestamp |
| Snowflake | ✅ | timestamp\_ltz |
| Oracle | ✅ | timestamp with time zone |
| SQL Server | ✅ | datetimeoffset |

---

# 四、最容易踩的 4 个坑（你做测试一定要覆盖）

---

## 🚨 坑1：DATETIME ≠ TIMESTAMP

```sql
DATETIME = "2024-04-13 10:00:00"

```

👉 不同用户理解不同时间

---

## 🚨 坑2：session timezone 影响结果

👉 同一条 SQL：

不同用户 → 不同结果 ❗

---

## 🚨 坑3：DATE + TIMESTAMP 混用

```sql
WHERE ts >= '2024-04-13'

```

👉 隐式变：

```text
2024-04-13 00:00:00（某个时区）

```
---

## 🚨 坑4：DST（夏令时）

👉 一天不是 24 小时

---

# 五、结合你们系统的关键建议

---

## ✅ 1. TemporalTool 必须补 timezone

```json
{
  "start": "2024-04-13",
  "end": "2024-04-13",
  "timezone": "Europe/Berlin"
}

```
---

## ✅ 2. Query 层必须统一

👉 不允许直接用日期字符串

---

## ✅ 3. 推荐统一策略（非常关键）

👉 内部统一：

```plaintext
全部转成 UTC TIMESTAMP

```
---

## 🧠 一句话总结

👉 数据库不是“是否支持时区”的问题，而是：

> ❗**你有没有在整个链路中保持“时间语义一致”**
