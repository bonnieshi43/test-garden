# Bug #74017 — AI Knowledge Base Management Features：测试分析报告

> 生成日期：2026-03-18
> 分析范围：KB 搜索、软删除/恢复/永久删除、审计日志、权限控制

---

## 第一部分：需求分析

### 核心目标

将 Knowledge Base Management 页面从基础 CRUD 工具升级为具备**搜索、权限控制、变更审计、误删恢复**能力的可信文档管理系统。

### 用户价值

| 需求编号 | 痛点 | 价值 |
|---------|------|------|
| #1 | 无法搜索 KB 文章，文档多了难以定位 | 快速找到目标文档以便更新 |
| #2 | 无法区分 AI 答案来源（标准文档 vs KB） | 验证 AI 回答来源，支持调试和质量保证 |
| #3 | 无法限制编辑/删除权限，无变更记录 | 权限分离，操作可追溯 |
| #4 | 误删后无法恢复 | 降低数据丢失风险 |
| #5 | 无版本历史/变更日志 | 审计文档生命周期 |

### 新增功能

- **KB 搜索**：文档列表关键字过滤 + Search KB Tab 语义搜索（仅搜索 KB 来源文档）
- **软删除与恢复**：删除移入垃圾桶，支持恢复或永久删除
- **审计日志**：所有操作均记录操作人和时间，提供文档级历史和全局日志视图

🔴 Note：
从代码看，完全没有实现多组织隔离, 从需求看暂时ignore

## 第二部分：变更分析

### 前端 UI 变更

页面从单一列表改为三 Tab 布局：

**Documents Tab**
- 新增关键字搜索框（实时过滤标题、内容、标签等）
- 新增"Show Deleted"开关（仅 Admin 可见），开启后列表包含已删除文档，已删除行以半透明 + "Deleted" 标签标识
- 行操作按钮随文档状态动态变化：
  - 正常文档：History / Edit / Delete / Upload（Delete 仅 Admin 可见）
  - 已删除文档：History / Restore / Permanently Delete（后两项仅 Admin 可见）
- 新增多选模式，支持批量删除（Admin）和批量上传
- Edit 弹窗新增创建人 / 最后修改人信息展示
- 三种删除操作各有专属确认弹窗（含取消按钮），永久删除弹窗有不可逆操作警告

**Search KB Tab（新增）**
- 输入关键词进行语义搜索，结果仅来自 KB 来源的文档
- 结果卡片展示：相关度评分（颜色区分高/中/低）、文档元信息标签、内容片段

**Activity Log Tab（新增，仅 Admin）**
- 展示所有操作的全局审计日志
- 支持关键词搜索、操作类型过滤、分页浏览

**History Dialog（新增）**
- 每行新增 History 图标，点击后弹窗展示该文档的操作时间轴（含加载中/有数据/加载失败/无记录四种状态）

### 新增 / 变更的 API 接口

| 接口 | 说明 | 权限 |
|---|---|---|
| `POST /documents/search` | KB 语义搜索 | 所有登录用户 |
| `GET /documents/:id/history` | 查询单篇文档的操作历史 | 所有登录用户 |
| `POST /documents/batch-delete` | 批量软删除 | 仅 Admin |
| `POST /documents/:id/restore` | 恢复软删除文档 | 仅 Admin |
| `DELETE /documents/:id/permanent` | 永久删除（需先软删除） | 仅 Admin |
| `GET /documents/audit-log` | 查询全局审计日志 | 仅 Admin |
| `GET /documents?includeDeleted=true` | 文档列表含已删除项 | 仅 Admin |
| `DELETE /documents/:id` | 原为永久删除，**改为软删除** | 仅 Admin（原为所有用户） |

### 关键行为变化对比

| 之前 | 之后 | 测试关注点 |
|---|---|---|
| 删除 = 立即从系统中永久移除 | 删除 = 移入垃圾桶，可恢复；永久删除需二次操作 | 软删除后文档是否仍存在于数据库；KB 搜索是否已看不到该文档 |
| 新上传文档与历史文档在 KB 搜索中无区别 | 只有此次更新后重新上传的文档才会出现在 KB 搜索中 | 历史存量文档在 KB 搜索中不可见，需重新上传 |
| 任何登录用户均可删除文档 | 删除/恢复/永久删除仅 Admin 可操作 | 非 Admin 调用删除接口应返回 403 |
| 无操作记录 | 所有操作均写入审计日志 | 每种操作后审计日志是否有对应记录 |
| 删除操作点击即执行，无确认 | 三种删除各有确认弹窗 | 弹窗内容和取消行为是否正确 |
| 操作按钮固定不变 | 已删除行按钮切换为 Restore / Permanently Delete | 不同状态下按钮组是否正确切换 |
| 无历史记录入口 | 每行新增 History 图标，打开时间轴弹窗 | History 弹窗四种显示状态是否正确 |

### 需求覆盖度

| 需求 | 状态 | 说明 |
|---|---|---|
| #1 搜索 KB 文章 | ✅ 已覆盖 | Documents Tab 关键字过滤；Search KB Tab 语义搜索 |
| #2 仅搜索 KB 来源文档 | ✅ 已覆盖 | KB 搜索结果严格过滤，标准文档不会出现；**注意**：此次更新前上传的历史文档不会出现在搜索结果中，需重新上传 |
| #3 权限控制 + 记录操作者 | ⚠️ 部分覆盖 | 删除/恢复/永久删除有 Admin 限制；**创建和编辑对所有登录用户开放**；操作者信息记录在审计日志中 |
| #4 误删恢复 | ✅ 已覆盖 | 软删除 → 垃圾桶 → Restore 流程完整 |
| #5 版本历史 / 审计日志 | ✅ 已覆盖 | 6 类操作均有审计记录；文档级 History 弹窗 + 全局 Activity Log Tab |

## 第三部分：风险识别

| # | 类型 | 描述 | 严重度 |
|---|---|---|---|
| R1 | **数据一致性** | 删除时先从 KB 搜索中移除向量，再标记数据库软删除。若向量已移除但数据库标记失败，会出现文档在列表中显示正常、但 KB 搜索中已消失的矛盾状态 | 高 |
| R2 | **数据一致性** | 此次更新前已上传的历史文档不会出现在 KB 搜索结果中，页面仅有一行小字提示，用户可能误以为 KB 内容不完整 | 中 |
| R3 | **功能** | 审计日志写入失败时不会影响主操作流程，也不会有任何提示。关键操作（如删除、恢复）可能静默漏记，合规场景下无法察觉 | 中 |
| R4 | **功能** | 恢复文档后，文档在列表中可见，但 KB 搜索中不可用（删除时已同步移除向量），状态显示为"Needs Re-upload"。用户需手动重新上传才能恢复搜索可用性，当前界面对此无明确引导 | 中 |
| R5 | **功能** | 开启"Show Deleted"后，已删除文档的 Edit 按钮仍可点击，但保存时会静默失败（后端拒绝更新已删除文档）。用户无法从 UI 判断操作为何失败 | 中 |
| R6 | **兼容性** | 历史文档重新上传时，若旧的向量数据未被正确覆盖，可能在 KB 中残留重复的旧数据 | 中 |
| R7 | **权限** | 审计日志接口依赖全局认证中间件保护，若中间件注册顺序有误，该接口可能被未登录用户访问 | 低-中 |
| R8 | **UI 界面** | Activity Log Tab 仅对 Admin 渲染。非 Admin 用户若通过其他方式触发切换到该 Tab，页面内容为空白且无任何提示 | 低 |
| R9 | **UI 界面** | 多选模式下"Select All"仅选中当前页的文档，而非全部过滤结果。用户在搜索/过滤后跨页操作时，可能误以为已全选所有匹配文档 | 低-中 |

## 第四部分：测试策略

### 核心关注点

1. **软删除完整性**：删除 → 垃圾桶 → 恢复 / 永久删除的完整流程，数据库状态与 KB 搜索状态是否一致
2. **KB 搜索隔离性**：搜索结果是否仅来自 KB 来源文档；历史存量文档是否按预期不出现
3. **权限边界**：非 Admin 调用受保护接口是否返回 403；UI 层是否同步隐藏相关按钮和开关
4. **审计完整性**：6 种操作（创建/编辑/上传/删除/恢复/永久删除）是否均有对应记录
5. **状态一致性**：恢复后文档状态是否为"Needs Re-upload"；编辑已删除文档是否被正确拒绝
6. **文档列表 UI**：关键字搜索、过滤、排序在"Show Deleted"模式下是否正常；已删除行外观和按钮组是否正确切换
7. **跨系统一致性**：删除后 KB 搜索中向量是否同步移除；恢复后 KB 搜索中是否不可用（直至重新上传）
8. **UI 渲染完整性**：三种确认弹窗文案和取消行为；History 弹窗四种状态；Search KB 结果卡片格式；Select All 范围（R9）

### Scenario 1：操作按钮随文档状态和用户角色正确显示

**目的**：系统性验证每行操作按钮在不同状态和角色组合下的显示是否正确。

**步骤**：
1. Admin 用户，正常文档行，观察操作按钮
2. 非 Admin 用户，正常文档行，观察操作按钮
3. Admin 用户，开启"Show Deleted"，观察已删除文档行的操作按钮
4. 任意用户进入多选模式，观察 History 和 Edit 按钮是否变为不可点击
5. 找到状态为"Uploaded"的文档，观察 Upload 按钮状态
6. 找到状态为"Needs Re-upload"的文档，观察 Upload 按钮颜色

**预期结果**：

| 情形 | 应显示的按钮 | 不应显示 / 应禁用 |
|------|------------|----------------|
| Admin + 正常文档 | History、Edit、Delete、Upload | — |
| 非 Admin + 正常文档 | History、Edit、Upload | 无 Delete |

| 状态为"Uploaded" | — | Upload 按钮不可点击 |
| 状态为"Needs Re-upload" | Upload 按钮显示橙色警告色 | — |
| Select All（超过 10 条结果时翻至第 2 页）| 仅当前页文档被选中 | 第 1 页文档未被选中 |

**覆盖风险**：R5（已删除文档操作控制）、R9（Select All 范围）

🔴 **测试结果**：Pass

### Scenario 1：软删除后文档从列表消失、在垃圾桶可见

**目的**：验证软删除将文档移入垃圾桶，而不是永久删除，且 UI 与数据状态一致。

**前置条件**：已有一篇状态为"Uploaded"的文档。

**步骤**：
1. Admin 用户（dev-user）登录，进入 Knowledge Base → Documents Tab
2. 点击某文档行的 Delete 图标（tooltip 显示"Move to Trash"）
3. 确认弹窗出现，检查弹窗内容后点击"Move to Trash"
4. 观察文档列表（"Show Deleted"关闭状态）
5. 开启"Show Deleted"开关，观察该文档的行外观
6. 切换到 Search KB Tab，搜索该文档相关内容
7. 进入 Activity Log Tab，查找该文档对应的删除记录

**预期结果**：
- 步骤 3：弹窗标题为"Move to Trash"，正文说明文档将被移入垃圾桶且可由管理员恢复；点击"Cancel"可关闭弹窗，文档无任何变化
- 步骤 4：文档不再出现在列表中；页面底部提示"Document moved to trash"
- 步骤 5：文档重新出现，行整体变为半透明，标题旁显示红色"Deleted"标签；操作列变为 History / Restore / Permanently Delete，Edit 和 Upload 图标消失
- 步骤 6：KB 搜索中找不到该文档
- 步骤 7：Activity Log 中出现该文档的"Moved to Trash"记录，操作人为当前登录用户，操作时间为刚才执行删除的时间

**覆盖风险**：R1（数据一致性）

🔴 **测试结果**：Pass

### Scenario 2：恢复后文档可见，但需重新上传才能在 KB 搜索中出现

**目的**：验证恢复操作的状态正确性，以及恢复后 KB 搜索不可用的预期行为。

**前置条件**：已有一篇处于软删除状态的文档。

**步骤**：
1. Admin 开启"Show Deleted"，找到一篇已删除文档，点击 Restore 图标
2. 关闭"Show Deleted"，检查文档是否重新出现在列表中
3. 查看该文档的状态标签

5. 在 Documents Tab 对该文档点击 Upload，重新上传到 KB
6. 再次在 Search KB Tab 搜索

**预期结果**：
- 步骤 2：文档重新出现在正常列表中；页面底部提示"Document restored"
- 步骤 3：文档状态显示为"Needs Re-upload"
- 步骤 4：Search KB 中找不到该文档
- 步骤 6：Search KB 中可以找到该文档，显示相关度评分

**覆盖风险**：R4（恢复后搜索不可用）

🔴 **测试结果**：Pass

### Scenario 3：永久删除须先移入垃圾桶

**目的**：验证系统阻止直接永久删除正常文档，且永久删除确认弹窗有明确的不可逆警告。

**前置条件**：已有一篇正常状态的文档和一篇已软删除的文档。

**步骤**：
1. 对一篇**正常状态**的文档，直接调用永久删除接口（`DELETE /documents/:id/permanent`）
2. 观察接口响应
3. 通过 UI 对该文档先执行软删除（移入垃圾桶）
4. 开启"Show Deleted"，找到该文档，点击 Permanently Delete 图标
5. 检查弹窗内容后确认执行
6. 开启"Show Deleted"，确认该文档在列表中彻底消失；进入 Activity Log Tab，查找该文档对应的永久删除记录

**预期结果**：
- 步骤 2：接口返回 409 错误，提示文档必须先移入垃圾桶才能永久删除
- 步骤 4：弹窗顶部显示红色不可逆操作警告（"This action cannot be undone."），正文中包含文档标题，确认按钮为红色"Delete Permanently"；点击"Cancel"可关闭弹窗无任何变化
- 步骤 5：文档从列表消失；页面底部提示"Document permanently deleted"
- 步骤 6：即使开启"Show Deleted"，列表中也找不到该文档（区别于软删除，永久删除后不可恢复）；Activity Log 中出现该文档的"Permanently Deleted"记录，含操作人和操作时间

**覆盖风险**：R3（审计完整性）

🔴 **测试结果**：Pass

### Scenario 5：KB 语义搜索仅返回 KB 来源文档，结果卡片格式正确

**目的**：验证 Search KB 搜索结果只包含 KB 文档（不含标准文档），以及结果卡片的展示格式。

> **背景说明（解释"搜索任意词都能返回文档"的原因）**
>
> Search KB 底层是**向量语义搜索**，工作机制是：计算查询词与所有 KB 文档向量的相似度，然后返回相似度最高的 top-K 个结果（默认最多返回top10）。
> **关键点：系统没有相关度阈值过滤，只要 KB 中有文档，搜索任何词都会返回结果。**
> 因此，当 KB 中只有 3 篇文档时，搜索任意字符（哪怕是"abc"或随机乱码）也会把全部 3 篇都返回——这是**正常行为**，而非 bug。
> 测试的关注点不是"无关词是否返回空结果"，而是：
> 1. **相关度评分是否反映真实语义相关性**（相关词搜出来的文档评分应显著高于随机词搜出来的）
> 2. **结果来源是否严格限定为 KB 文档**（标准文档不会出现）
> 3. **KB 为空时才会显示"无结果"提示**

**前置条件**：
- KB 中已有至少 3 篇上传成功的 KB 文档（且内容各不相同，有明显的主题差异）
- 系统中同时存在标准文档和 KB 文档（用于验证来源隔离）

**步骤**：
1. 切换到 Search KB Tab，不输入任何内容，观察 Search 按钮状态
2. 输入与某篇**标准文档**（非 KB）强相关的关键词，按 Enter 键触发搜索，观察结果来源
3. 输入与某篇 **KB 文档**内容高度相关的关键词（例如该文档的标题词），点击 Search 按钮，观察搜索过程中按钮状态
4. 搜索完成后，查看结果卡片的展示内容；记录最高分文档的评分颜色
5. 输入一段与所有 KB 文档**语义无关**的随机字符（如"zzz123"），再次搜索，观察返回结果数量和评分
6. 对比步骤 4 与步骤 5 中同一篇文档的评分差异
7. 验证空 KB 场景：如果条件允许，临时软删除所有 KB 文档后搜索，观察空结果提示（测试完成后恢复）

**预期结果**：
- 步骤 1：Search 按钮为禁用状态，无法点击
- 步骤 2：结果中**不出现标准文档**（即使内容语义相关）；按 Enter 可正常触发搜索
- 步骤 3：搜索过程中 Search 按钮内显示加载动画，不可重复点击
- 步骤 4：顶部显示"N results from KB"；每张结果卡片包含：
  - 文档标题 + 相关度评分（与查询词语义相关的文档评分高，显示绿色；一般相关显示橙色；低相关显示灰色）
  - 模块 / 上下文类型 / 主题领域等元信息标签
  - 文档内容片段（过长时截断并显示"…"）
  - 右上角 Edit 图标，点击后跳转至 Documents Tab 并打开编辑弹窗
- 步骤 5：**3 篇文档仍然全部返回**（这是正常行为），但所有文档的评分均明显低于步骤 4，颜色整体偏橙色或灰色
- 步骤 6：同一篇文档在"相关词搜索"与"随机词搜索"下评分可见差异，说明评分反映了真实语义相关度
- 步骤 7：KB 中没有任何有效文档时，显示"No KB articles matched your query."

**覆盖风险**：R2（历史文档隔离）

🔴 **测试结果**：Pass
