# 文档切割规则（简化版）

> 本文是面向“使用者 / 产品 /文档团队”的简版说明，只保留关键规则，不展开源码细节。

---

## 一、文档类型与整体策略

- **普通文档（大部分模块）**
  - 按模块 + 文件大小决定是否切割。
  - 切割后按文档结构（H2、H3 标题）分块。
  - 每个块前面会自动加上该块的“摘要信息”和文档上下文。

- **Web API 文档（`webapi` 模块）**
  - 不做任何切割。
  - 整个 `.md` 文件作为一个块，仅在前面加统一的 context 说明。

---

## 二、是否切割：按模块 + 文件大小

### 2.1 特殊模块（脚本 / 图表 API）

- 模块名包含以下任意关键字：
  - `chartAPI`
  - `commonscript`
  - `viewsheetscript`
- 规则：
  - 文件大小 `< 5KB` → **不切割**（整体作为一个块）。
  - 文件大小 `≥ 5KB` → **按 H2 / H3 切割**。

### 2.2 其他模块

- 所有非上述三种模块。
- 规则：
  - 文件大小 `< 3KB` → **不切割**。
  - 文件大小 `≥ 3KB` → **按 H2 / H3 切割**。

---

## 三、分块方式

### 3.1 一级切割：按 H2 标题

- 当“需要切割”时，先按 Markdown 的 H2 标题（`##`）分块。
- 每个块的内容结构：

  > H1 标题（如存在）  
  > + H1 与第一个 H2 之间的“引言内容”（intro）  
  > + 某一个 H2 标题  
  > + 该 H2 下的全部内容

- 同一文档中，每个 H2 块都会共享同一个 H1 + 引言内容，保证每块都有基本上下文。

- 特殊情况：
  - 如果文档里没有 H2 标题 → 整篇文档不拆分。
  - 如果只有 H1 → 整篇文档作为一个块。

### 3.2 二级切割：按 H3 标题

- 在 H2 块基础上，如果**满足下列条件**，才会进一步按 H3（`###`）切割：
  1. H2 块里确实存在 H3 标题。
  2. 该 H2 块的正文内容足够长（估算 token 数大于约 **1000**）。

- 满足条件时：
  - 每个正常的 H3 标题会成为一个新的小块：
    - 保留同一个 H1 + 引言 + 对应 H2 标题。
    - 再加上这个 H3 标题和它的内容。
  - 标题为 `Parameters` 或 `Examples` 的 H3 不单独成块：
    - 会自动合并到前一个相关块中。

- 如果 H2 块不够长 / 没有 H3：
  - 保持整个 H2 块不再拆分。

---

## 四、块内容中包含的信息

对用户来说，每个最终“块”（chunk）里大致包含：

1. **文档级头信息**
   - 文档名（模块 + 文档名）。
   - 文档在目录树中的位置（上下文描述）。
   - 文档所属的“使用场景”和“主题标签”（subjects）。

2. **自动生成的摘要**
   - 整体 / H1 摘要。
   - 对应 H2 摘要。
   - 若按 H3 拆分，则是当前 H3 摘要；否则是该 H2 下所有 H3 的摘要合并。

3. **当前块的正文内容**
   - 对应的 H2 / H3 标题及其内容。
   - 去掉内部用于 AI 的辅助标记（不会影响阅读）。

> 结果是：每个块既有总结（方便检索），又有相对完整的局部原文（方便回答具体问题）。

---

## 五、大块（超长块）记录规则

- 系统会对每个最终块估算长度（以 token 粗略度量，约等于“字数 / 4”）。
- 如果一个块估算长度 **大于约 1200 token**：
  - 会被视为“大块”，并记录到 `large-chunks-analysis.json` 文件中，用于后续分析和优化。
- 该记录信息包括：
  - 模块名、文档名、文档路径。
  - 块序号、估算 token 数、字节大小。
  - 块内容前 200 个字符的预览。

> 这一机制只是“监控和调优用”，不会影响用户问答本身。

---

## 六、Web API 文档的特殊规则（只读，不切）

- 模块名：`webapi`。
- 不执行任何 H2 / H3 切割。
- 每个 `.md` 文档直接作为一个块，前面统一加上：
  - `Usage context is: REST API`
  - `Subject area is: REST API`
- 链接（`docUrl`）通过预生成的映射文件确定，未匹配到时使用默认文档 URL。

---

## 七、如何编写更“友好”的文档以适配切割

- 建议：
  - 合理使用 `#` / `##` / `###` 标题层级。
  - 确保每个 H2 小节内容相对独立、主题明确。
  - 对特别长的 H2 小节，可以用多个 H3 拆成子场景。
  - `Parameters` / `Examples` 放在对应主段落下，系统会自动和主块合并，不会被单独拆出。
