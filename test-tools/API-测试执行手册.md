# 通用 API 测试执行手册

## 目标

本文档用于沉淀一套可复用的 API 测试流程，适合作为后续所有 API 功能测试、权限测试、回归测试、安全校验测试的统一参考模板。

适用场景包括但不限于：

- 新增 API 功能验证
- Bug 修复回归验证
- 权限控制校验
- 参数边界与异常输入测试
- UI 与 API 联动行为验证
- 服务端“是否真正生效”的持久化验证

本文档强调一个核心原则：

**不要只看接口是否返回 200，还要验证服务端是否真正执行了预期逻辑。**

---

## 一、测试设计原则

在执行 API 测试前，先明确以下几点：

1. **测试目标是什么**
   - 要验证新增功能？
   - 要复现某个 bug？
   - 要确认权限限制是否生效？
   - 要确认非法输入是否被拒绝？

2. **接口的真实行为应该是什么**
   - 成功时应返回什么？
   - 失败时应返回什么？
   - 拒绝时应返回错误，还是静默忽略？
   - 是否会影响数据库、任务配置、权限状态或其他持久化结果？

3. **接口属于哪一类测试**
   - 查询类（GET / read）
   - 创建类（POST / create）
   - 更新类（PUT / PATCH / save）
   - 删除类（DELETE）
   - 权限敏感类
   - 依赖浏览器上下文类（cookie / xsrf / referer）

如果接口是更新类或权限敏感类，通常必须增加“读回验证”步骤，不能只看响应码。

---

## 二、测试前置条件

开始测试前，请确认以下信息已经准备好。

### 1. 服务与环境

- 测试环境地址：`<BASE_URL>`
- 接口路径：`<API_PATH>`
- 服务已启动且可访问
- 如果接口区分环境，明确当前测试环境：`dev / qa / staging / local`

### 2. 账号与权限

准备至少以下测试账号：

- **正常账号**：具备接口所需权限
- **受限账号**：缺少目标权限，用于验证拒绝逻辑
- **无效账号**（可选）：用于验证未登录、鉴权失败或越权场景

如果是权限测试，要事先明确：

- 目标权限名称
- 权限所在模块
- 权限是允许还是显式拒绝
- 权限变更后，UI 侧是否也同步隐藏/禁用对应能力

### 3. 测试数据

根据接口类型准备必要数据：

- 已存在的实体或记录
- 新建请求所需的最小合法 payload
- 非法 payload
- 边界值 payload
- 具备真实业务结构的 payload

建议优先使用**从真实请求中复制的 payload**，避免手工臆造复杂结构。

### 4. 请求上下文

明确该接口调用时需要哪些上下文：

- Bearer Token / API Token
- Session Cookie
- XSRF Token
- `X-Requested-With`
- `Origin`
- `Referer`
- 特定租户、组织、owner、locale、timezone 信息

有些接口虽然表面是 API，但实际上依赖浏览器登录态。遇到这种接口时，应尽量模拟真实浏览器请求。

---

## 三、测试流程

以下流程可作为所有 API 测试的标准步骤。

### 第 1 步：明确测试点与预期结果

先写清楚本次验证的问题：

- 接口名称：`<METHOD> <API_PATH>`
- 测试目标：`<功能 / 权限 / bug / 回归>`
- 成功预期：`<预期返回 + 预期状态变化>`
- 失败预期：`<预期错误码 + 预期错误信息>`

建议至少定义以下三类预期：

#### 成功预期
- 请求成功
- 返回码正确
- 返回体符合预期
- 服务端状态发生了正确变化

#### 明显失败特征
- 请求返回成功，但结果不正确
- 不应通过的请求被放行
- 不应写入的数据被真正持久化

#### 修复不完整特征
- 请求返回成功
- 服务端未真正执行目标操作
- 或服务端静默忽略非法字段，但没有显式报错

---

### 第 2 步：从真实浏览器请求中获取认证信息与基线请求

对于大多数复杂 API，尤其是依赖登录态、权限、cookie、xsrf 或页面上下文的接口，推荐**直接从浏览器复制真实请求**，一次性拿到：

- 请求 URL
- 请求 Method
- 请求 Headers
- 认证信息（token / cookie / xsrf）
- 请求体 payload

推荐操作方式：

1. 打开对应页面
2. 按 `F12`
3. 进入 `Network`
4. 在页面中正常完成一次目标操作
5. 找到对应请求
6. 右键该请求，选择：`Copy` > `Copy as cURL (bash)`

### 为什么推荐这种方式

相比手工拼 header 和 payload，这种方式更稳定：

- 能直接复用真实登录态
- 不容易漏掉关键请求头
- 不容易把复杂 JSON 改坏
- 特别适合权限类、保存类、配置类、复杂表单类接口

如果接口本身很简单，只依赖纯 token，也可以不走浏览器复制方式，直接通过登录接口换取 token 后构造请求。

#### 补充：仅通过登录接口换取 token 的常见方式

```bash
curl -X POST "<LOGIN_URL>" \
  -H "Content-Type: application/json" \
  -d '{"username":"<USERNAME>","password":"<PASSWORD>"}'
```

可提取的信息包括：

- `token`
- `access_token`
- `refresh_token`（如有）
- Session 信息（如有）

但要注意：

**某些接口仅有 token 仍然不够，必须同时带上 cookie 与 xsrf 相关请求头。**

---

### 第 3 步：先确认 UI / 权限前置行为

如果接口与页面能力、权限控制、按钮显隐有关，建议先在 UI 验证一遍前置状态：

- 用户是否能看到该入口
- 被限制的功能是否被隐藏或禁用
- 正常用户与受限用户表现是否不同

这一步的目的不是替代 API 测试，而是帮助确认：

- 前端限制是否存在
- 接下来要验证的是不是“后端是否也做了同样限制”

---

### 第 4 步：整理测试输入

在第 2 步复制得到的真实请求基础上构造测试样例。

建议至少准备以下几类 payload：

#### 1. 基线合法请求
用于验证接口在正常条件下可成功执行。

#### 2. 目标测试请求
只修改与你本次验证相关的字段。

例如：

- 插入受限字段
- 修改边界值
- 删除必填字段
- 传入非法枚举
- 构造越权对象

#### 3. 异常请求
用于验证接口对错误输入的健壮性：

- 缺字段
- 字段类型错误
- 空数组 / 空字符串 / null
- 超长值
- 非法组合

最佳实践：

**不要从零手写复杂 JSON，优先复制真实 payload 后做最小修改。**

如果是从浏览器 `copy as curl(bash)` 开始整理，推荐先把 `--data-raw` 抽成独立的 json 文件，再在这个文件基础上做差异修改。

---

### 第 5 步：整理完整请求头与执行命令

如果你已经通过浏览器 `Copy as cURL (bash)` 获取到了原始请求，那么这一阶段的目标是：

1. 保留必要请求头
2. 删除无关或噪音头（如有）
3. 将 `--data-raw` 改为引用本地 json 文件
4. 把 bash 版本转换成 Windows `cmd` 可直接执行的 `curl.exe` 命令

### 推荐的 AI 辅助整理方式

把第 2 步从浏览器复制出来的 `curl(bash)` 原始内容直接粘贴给 AI，并输入下面这段提示词：

```text
请把下面这个从浏览器 copy as curl(bash) 得到的命令整理成适合 Windows cmd 直接运行的版本：
1. 将 --data-raw 的内容单独提取出来，保存成一个 json 文件
2. curl 命令改写成 cmd 可直接执行的格式
3. 保留必要的请求头，不要丢掉 cookie、xsrf、authorization、origin、referer 等关键信息
```

AI 整理后的目标结果应当包括两部分：

#### 1. 一个独立的 payload json 文件
例如：

- `C:\Users\Administrator\payload.json`

#### 2. 一个可直接在 Windows `cmd` 中执行的 `curl.exe` 命令

例如：

```bash
curl.exe -i -X POST "<FULL_URL>" ^
  -H "Content-Type: application/json" ^
  -H "Cookie: <COOKIE>" ^
  -H "X-XSRF-Token: <TOKEN>" ^
  --data-binary "@C:\Users\Administrator\payload.json"
```

---

### 第 6 步：发送请求并记录原始响应

使用 curl、Postman、IDE HTTP Client 或脚本发送请求。

#### 通用 curl 模板

```bash
curl.exe -i -X <METHOD> "<FULL_URL>" ^
  -H "Content-Type: application/json" ^
  -H "Authorization: Bearer <TOKEN>" ^
  --data-binary "@<PAYLOAD_FILE>"
```

如果需要完整浏览器上下文，则在此基础上补齐 cookie / xsrf / referer 等头。

执行后请至少记录：

- HTTP 状态码
- 响应头
- 响应体
- 请求时间
- 触发环境
- 使用账号

如果只想快速看状态码，可使用：

```bash
curl.exe -s -o NUL -w "HTTP %{http_code}\n" -X <METHOD> "<FULL_URL>"
```

---

### 第 7 步：做结果验证，而不只看状态码

这是通用 API 测试里最关键的一步。

根据接口类型进行二次验证。

#### 对查询类接口
验证：
- 返回字段是否齐全
- 数据是否正确
- 筛选 / 排序 / 分页是否正确

#### 对创建类接口
验证：
- 返回是否包含新对象 ID
- 新对象是否真的创建成功
- 后续查询接口是否能读到

#### 对更新类接口
验证：
- 修改后的字段是否真的生效
- 未修改字段是否被错误覆盖
- 再次读取对象时结果是否与预期一致

#### 对删除类接口
验证：
- 目标数据是否真的删除
- 再查时是否返回不存在
- 是否存在软删除 / 异步删除逻辑

#### 对权限类接口
验证：
- 无权限请求是否被明确拒绝
- 是否只是 UI 限制而后端未拦截
- 被禁止的字段是否被偷偷保存
- 是否存在“返回 200 但静默忽略”的情况

常见验证方式：

- 立即调用读取接口进行读回确认
- 在 UI 中刷新后观察结果
- 查询数据库 / 日志 / 管理后台
- 通过后续流程验证副作用是否存在
